# ProxyAccessHub

## Назначение проекта

ProxyAccessHub — веб-приложение для управления доступом пользователей к proxy-сервису. Проект предоставляет пользовательские и административные сценарии для создания подключений, продления доступа, отображения данных по пользователям и обработки статусов синхронизации.

## Архитектура решения

Решение разделено на несколько слоёв:

- `ProxyAccessHub` — ASP.NET Core Razor Pages приложение, содержащее точки входа, страницы пользовательского интерфейса, авторизацию и конфигурацию HTTP-конвейера.
- `ProxyAccessHub.Application` — слой прикладной логики, в котором размещены сервисы сценариев, контракты, модели и конфигурация бизнес-операций.
- `ProxyAccessHub.Domain` — доменные сущности, перечисления и value object-объекты предметной области.
- `ProxyAccessHub.Infrastructure` — инфраструктурный слой для работы с базой данных, внешними API, фоновой синхронизацией и реализациями репозиториев.

## Основные сценарии

- Создание нового подключения пользователя с последующей выдачей proxy-ссылки.
- Поиск пользователя по `telemt id` или данным proxy-ссылки для продления доступа.
- Формирование и обработка платёжных сценариев.
- Административный просмотр списка пользователей и состояния синхронизации.
- Фоновая синхронизация локальных пользователей с внешней системой telemt.

## Внешние зависимости и интеграции

- `telemt API` используется для получения списка пользователей, создания новых пользователей и синхронизации состояния доступа.
- `SQL Server` используется как основное хранилище данных приложения.
- `YooMoney` используется в сценариях уведомлений и обработки оплаты.
- `Serilog` используется для логирования приложения.

## Особенности текущей реализации

- Приложение использует Razor Pages как основной UI-подход.
- Фоновая синхронизация пользователей запускается через `BackgroundService`.
- Доступ к данным построен через `UnitOfWork` и специализированные репозитории.
- В решении уже присутствуют миграции базы данных и модели хранения для пользователей, серверов, подписок и платежей.

## Что важно учитывать при изменениях

- Основная бизнес-логика должна оставаться в слоях `Application` и `Domain`, а инфраструктурные детали — в `Infrastructure`.
- Изменения, затрагивающие синхронизацию с telemt, нужно проверять вместе со связанными сценариями создания пользователя и отображения статуса в административной части.
- Изменения платёжных сценариев следует согласовывать с обработкой уведомлений и пользовательскими страницами продления и создания доступа.
- Для страницы `ProxyAccessHub/Pages/Admin/Payments.cshtml` колонка статуса должна использовать только варианты `Оплачено`, `Ожидает` и `Переплата`; статусы ручной обработки на этой странице не отображаются.
- Для страницы `ProxyAccessHub/Pages/Admin/Payments.cshtml` модальное окно проверки платежа не должно содержать блок `Сводка по заявке`; в нём показывается один список платежей из базы данных без отдельной таблицы платежей YooMoney.
- Для страницы `ProxyAccessHub/Pages/Admin/Payments.cshtml` кнопка получения данных из YooMoney должна автоматически загружать отсутствующие операции, сохранять их в базу данных и сразу применять их к пользователю без дополнительного ручного шага.
- Для страницы `ProxyAccessHub/Pages/Admin/Payments.cshtml` в списке платежей заявки нужно показывать и сумму применения, и фактическую сумму поступления; фактическая сумма хранится отдельно и отображается рядом с суммой заявки.
- Для страницы `ProxyAccessHub/Pages/Admin/Payments.cshtml` при чтении `operation-history` показывать администратору фактическую сумму из поля `amount`, а при автоматическом применении операции использовать сумму локальной заявки, чтобы комиссия не перекладывалась на пользователя.
- Для интерфейса по умолчанию использовать Bootstrap и его utility-классы из локальных файлов `wwwroot/lib`. Не подключать Bootstrap через CDN и не добавлять новый локальный CSS, если задачу можно решить средствами Bootstrap.
- Для быстрой проверки корректности сборки проекта достаточно `dotnet build`, если пользователь не просил большего.

## Дополнительные локальные правила

- При проверке связи с сервером telemt учитывать, что в поле `Host` может быть указан как просто хост, так и `host:port` или полный URL со схемой и портом.
- Если порт в `Host` не указан, для telemt API по умолчанию использовать порт `9091`.
- Проверку связи с сервером telemt выполнять не через ICMP/Ping, а реальным HTTP-запросом к telemt API с учётом `Authorization`-заголовка из конфигурации.
- Для telemt API учитывать, что bearer-токен передаётся точным значением заголовка `Authorization`, например `Bearer ...`.
- По локальному исходному коду telemt можно ориентироваться на каталог `D:\Media\User\Downloads\telemt-main`, если нужно уточнить поведение API или конфигурации.
- В корне проекта есть папка `Documentation` с файлами документации по API telemt и YooMoney; при задачах по интеграции сначала проверять эти локальные документы.
